# US-003.6 - Step Out Execution

## Story

As an MSX-BASIC developer, I want to exit the current function so I can return to the calling scope.

## Acceptance Criteria

- Stepping Out works as expected.

## BDD Scenarios

### Scenario 1: Step Out from a nested function

Given the program is paused inside a function (not at root scope)
And the current call stack depth is greater than one
When the user executes Step Out
Then all breakpoints must be disabled
And the return address must be retrieved using peek16 [reg SP]
And a temporary breakpoint must be created at the return address if no breakpoint exists there
And the emulator must continue execution (debug cont)
And execution must pause at the return address
And all original breakpoints must be re-enabled
And the call stack depth must decrease by one

### Scenario 2: Step Out when return address already has a breakpoint

Given the program is paused inside a function
And the return address already has an existing breakpoint
When the user executes Step Out
Then all breakpoints must be disabled
And the return address must be retrieved using peek16 [reg SP]
And the existing breakpoint at the return address must be enabled
And no temporary breakpoint should be created
And the emulator must continue execution (debug cont)
And execution must pause at the return address
And all breakpoints must be re-enabled

### Scenario 3: Step Out from root scope

Given the program is paused at the root scope (call stack depth is one)
When the user executes Step Out
Then continueRequest action must be performed

### Scenario 4: Step Out with no breakpoints defined

Given the program is paused inside a function
And there are no breakpoints defined
When the user executes Step Out
Then the return address must be retrieved using peek16 [reg SP]
And a temporary breakpoint must be created at the return address
And the emulator must continue execution (debug cont)
And execution must pause at the return address
And no error should occur

### Scenario 5: Step Out with disabled breakpoints

Given the program is paused inside a function
And there are breakpoints defined but all are disabled
When the user executes Step Out
Then all breakpoints must remain disabled during execution
And the return address must be retrieved using peek16 [reg SP]
And a temporary breakpoint must be created if needed
And the emulator must continue execution (debug cont)
And execution must pause at the return address
And all breakpoints must be re-enabled after the pause

### Scenario 6: Step Out restores breakpoint state after execution

Given the program is paused inside a function
And there are multiple breakpoints with mixed enabled/disabled states
When the user executes Step Out
Then all breakpoints must be disabled before continuing
And execution must continue until the return address
And after pausing, all breakpoints must be restored to their original states

### Scenario 7: Step Out in deeply nested calls

Given the program is paused in a deeply nested function (call stack depth > 2)
When the user executes Step Out
Then execution must resume until the immediate caller's return address
And the call stack depth must decrease by exactly one
And subsequent Step Out commands must continue unwinding the stack one level at a time

## Technical specification

Use the following guidelines:

- Stepping out means going up in the scope of the call stack, when applicable;
- There's no stepping out from call stack root scope.

Procedure when stepping out:

1. If the stack is in the root scope, execute a continueRequest and skip all the steps below. Otherwise, continue with the steps below;
2. Disable all breakpoints;
3. Get the current stack returning address: `peek16 [reg SP]`;
4. If the current stack returning address is a existent breakpoint, enable it and go to step 6;
5. Otherwise, create a temporary breakpoint at current stack returning address: `debug breakpoint create -address [peek16 [reg SP]] -once 1`;
6. Continue emulation: `debug cont`;
7. When the breakpoint is hitted, enable all the breakpoints.
