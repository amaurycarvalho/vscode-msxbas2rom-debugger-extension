# US-003.1 - Stack Trace: Step Into

## Story

As an MSX-BASIC developer, I want to enter a function call on the current line so I can follow its execution line-by-line.

## Acceptance Criteria

- Stepping Into works as expected.

## BDD Scenarios

### Scenario 1: Step Into in a function call

Given the program is paused on a line containing a function call
And there are active breakpoints within the called function
When the user executes the Step Into command
Then all breakpoints should be enabled
And execution should continue (debug cont)
And execution should pause at the first breakpoint within the called function
And the call stack level should increase (entry into new scope)

### Scenario 2: Step Into on a line without a function call

Given the program is paused on a line that does not contain a function call
When the user executes the Step Into command
Then all breakpoints should be enabled
And execution should continue (debug cont)
And execution should pause at the next available breakpoint
And the call stack level should remain the same

### Scenario 3: Scenario 3: Step Into without defined breakpoints

Given the program is paused
And there are no Breakpoints defined in the program
When the user executes the Step Into command
Then the system should automatically enable breakpoints (if there is logic for it)
Or continue execution (debug cont)
And no error should occur
And the behavior should be equivalent to a Continue

### Scenario 4: Step Into in a function without an internal breakpoint

Given the program is paused on a line with a function call
And there are no breakpoints inside the called function
When the user executes the Step Into command
Then all breakpoints should be enabled
And execution should continue (debug cont)
And execution should pause at the next breakpoint outside the function
And no entry into the function scope should occur

### Scenario 5: Step Into with breakpoints disabled

Given the program is paused
And there are breakpoints defined, but disabled
When the user executes the Step Into command
Then all breakpoints should be enabled
And execution should continue (debug cont)
And execution should pause at the next valid breakpoint

### Scenario 6: Step Into with emulator not paused

Given the emulator is running (not paused)
When the user executes the Step Into command
Then the system should ignore the command or pause execution first
And no error should occur

### Scenario 7: Step Into in chained calls

Given the program is paused on a line with multiple chained function calls
When the user executes the Step Into command
Then execution should enter the first function call
And the call stack should correctly reflect the new depth
And subsequent Step Into executions should continue deepening the stack

### Scenario 8: Step Into maintains stack consistency

Given the program is paused within a function
When the user executes multiple Step Into commands
Then each step should correctly reflect the evolution of the call stack
And no scope inconsistency should occur

## Technical specification

Use the following guidelines:

- Stepping into means going down in the scope of the call stack, when applicable.

Procedure when stepping into:

1. Enable all breakpoints if they are not already active;
2. Continue the emulation until the next breakpoint: `debug cont`.
