## ADDED Requirements

### Requirement: MegaROM Image Launch
As an MSX-BASIC developer, I want the debugger to support MegaROM builds produced by MSXBAS2ROM so that I can debug larger programs that require bank switching. The debugger SHALL support launching MegaROM images produced by MSXBAS2ROM in a VSCode debug session using openMSX and SHALL support attaching to a running MegaROM program.

#### Scenario: Start debug session with MegaROM
- **WHEN** a MegaROM image generated by MSXBAS2ROM is selected
- **WHEN** a debug session is started in VSCode
- **THEN** openMSX launches with the MegaROM image loaded
- **THEN** the program starts under debugger control

#### Scenario: Attach to running MegaROM program
- **WHEN** a MegaROM program is already running in openMSX
- **WHEN** the debugger attaches to it
- **THEN** the debugger establishes control over the running program
- **THEN** source-level debugging becomes available

### Requirement: MegaROM ROM Format Detection
The debugger SHALL detect the MegaROM format from the ROM filename suffix. MSX-BASIC source code compiled in MegaROM format will have the ROM format embedded in the name suffix (e.g., `test10.bas` → `test10[ASCII8].rom` or `test10[KonamiSCC].rom`). The same applies to the .CDB file name.

#### Scenario: ROM format is detected from filename
- **WHEN** a MegaROM image is loaded
- **THEN** the ROM format (ASCII8, KonamiSCC, etc.) is detected from the filename suffix

### Requirement: CDB Address Format for MegaROM
The debugger SHALL interpret CDB addresses in 6-digit hexadecimal format, where the first two digits correspond to the MegaROM segment number and the last four digits correspond to the real address within the segment. Variable addresses SHALL use the last 4 digits (first two are zeros).

Concrete CDB example:
```
L:C$test53.bas$9$0$0
S:G$LIN_10$0_0$0
L:G$LIN_10$0_0$0:028490
```
In this example, `LIN_10` address is `0x8490` at MegaROM segment `0x02`.

Also, at example below, variable `A` address is 0xC038:
```
S:G$VAR_A$0_0$0({3}F24),G,0,0
L:G$VAR_A$0_0$0:00C038
```
Here variable `A` address is `0xC038` (segment prefix `00` discarded).

#### Scenario: Breakpoint address includes segment
- **WHEN** a breakpoint is set in a MegaROM program
- **THEN** the address is read as a 6-digit hex value with segment prefix

#### Scenario: Variable address discards segment
- **WHEN** a variable address is read from CDB for a MegaROM program
- **THEN** the first two digits (zeros) are discarded

### Requirement: Segment-Aware Breakpoints
The debugger SHALL create breakpoints in openMSX with segment conditions using `pc_in_slot` for MegaROM programs.

To create the breakpoint in the emulator, use the following template:
```
debug breakpoint create -address <address> -condition {[pc_in_slot X X <segment>]}
```

Concrete example:
```
debug breakpoint create -address 0x8490 -condition {[pc_in_slot X X 0x02]}
```

#### Scenario: Breakpoint created with segment condition
- **WHEN** a breakpoint is set in a MegaROM program
- **THEN** the breakpoint condition includes `pc_in_slot` for the correct segment

### Requirement: Active Segment Detection
The debugger SHALL read the current active MegaROM segment number from address `0xC023` (8-bit unsigned) using `peek 0xC023`.

#### Scenario: Active segment is read on breakpoint hit
- **WHEN** execution pauses at a breakpoint in a MegaROM program
- **THEN** the active segment is read from `0xC023`

### Requirement: MegaROM Stack Trace
The debugger SHALL handle MegaROM GOSUB calls which push 3 integers to the Z80 call stack: callback helper address, segment number, and return address.

#### Scenario: Stack trace reads 3 values per frame
- **WHEN** a stack trace is reconstructed in a MegaROM program
- **THEN** each GOSUB frame reads 3 consecutive values from the Z80 stack using `peek_u16 [reg SP]`

### Requirement: Plain ROM Backward Compatibility
MegaROM support SHALL NOT affect plain ROM debugging functionality.

#### Scenario: Plain ROM debugging works unchanged
- **WHEN** a plain ROM image is debugged
- **THEN** all existing debugging features continue to function as before
